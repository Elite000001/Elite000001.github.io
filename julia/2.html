Tutorial 2 - Type System and Multiple Dispatch

6 steps

In this tutorial, you will learn about Julia's type system and multiple dispatch. With multiple dispatch, you will learn how to write generic functions with multiple associated methods with different argument types.

You will learn the differences between an abstract type and a concrete type. Lastly, you will learn how to write your own custom abstract types as well as its subtypes.

Let's dive in and learn more about this beautiful language, Julia.

Helpful prior knowledge

Required to have finished Quest 1 or Tutorial 1.

Learning Outcomes

You will be able to:

  • Explain what is an abstract type vs a concrete types
  • Write generic functions with multiple associated methods
  • Build your own abstract types

Tutorial Steps

Total steps: 6

  • Step 1: Julia Base

    Before doing this step, ensure that you have finished Quest 1 / Tutorial 1. It contains instructions on how to set up your development environment.

    Let's start talking about the Julia standard library first. The Julia standard library is called Base.jl or just Base. It contains all the building blocks of the language including

    • Core functions and types that are essential to the language. 
    • Data types that defines numbers, strings and more i.e. Int, Float64, and String. 
    • Fundamental abstract types such as Number and AbstractString. 
    • Control structures such as loops, conditionals and exception handling. 
    • Built-in functions such as methods, methodswith, names, collect, and cumsum.

    Because Base.jl is built into Julia, its always available and doesnt require any external package installation. Its essentially the foundation upon which other Julia libraries and packages are built.

  • Step 2: Multiple Dispatch

    Multiple Dispatch is a core feature of the Julia programming language that allows a function to behave differently depending on the types of its arguments. This means that a single function name can have multiple method definitions, each corresponding to different combinations of argument types. The most appropriate method is chosen at runtime based on the types of the arguments provided to the function. Take for example the code below.

    function hello(x::Int)
  println("Hello, you are a $(x)!")
end

function hello(x::String)
  println("Hello, my name is $(x).")
end

function hello(x::Bool)
  if x
    println("Hello, you are so honest.")
  else
    println("Hello, you are a liar.")
  end
end
methods(hello)
    Plain text

    Run the code above to the julia> REPL. Your last output should show you that hello has 3 methods.

    # 3 methods for generic function "hello" from Main:
 [1] hello(x::Bool)
 	@ REPL[3]:1
 [2] hello(x::String)
 	@ REPL[2]:1
 [3] hello(x::Int64)
 	@ REPL[1]:1

    Run the following code to the julia> REPL.

    hello(10)
hello("Mark")
hello(false)
hello(1.0)
    Plain text

    Each will have a different output and the last one will error.

    Observe that because the code for the hello methods declares different argument types, each method can produce different outputs as well depending on their argument types.

    However, as the error implied, there is no method for the hello function to receive an argument type of Float64. To fix that, you can write a method for it. Here, write this one-liner for writing a function/method in Julia on the julia> REPL.

    hello(x::Float64) = println("Hello, you float like $(x).")
    Plain text

    You should be able to run the hello function with any 64-bit float such as 3.14, 1.5, and 10.5.

    Another advantage of multiple dispatch is it allows for highly efficient code because it can generate specialized code for different combinations of argument types at compile time. This is one of the reasons Julia can be both fast and expressive.

    Yeah we get it, but what even are abstract and concrete types? Let's move over to the next step to discuss what abstract and concrete types are.

  • Step 3: Abstract Types vs Concrete Types

    In Julia, an abstract type is a type that cannot be instantiated directly but serves as a general category or classification for other types. Abstract types allow you to define a hierarchy of types and provide a way to group related types together. They enable polymorphism and multiple dispatch by allowing you to define methods that operate on various types in a more flexible way.

    In Julia, a concrete type refers to a type that can have instances or objects created from it, as opposed to an abstract type. Concrete types are fully defined types that are capable of being instantiated and used directly to create values. These are typically the leaf nodes in the type hierarchy, meaning they do not have any subtypes (i.e., they are not further specialized).

    Another cool thing about concrete types is that they can be grouped together as part of an abstract type. Here is another example. Head to the help> REPL and type "Number" and press enter. It should tell you that it is an "Abstract supertype for all number types".

    Head to the julia> REPL then run the following code.

    subtypes(Number)
    Plain text

    It should give you the following output.

    3-element Vector{Any}:
 Base.MultiplicativeInverses.MultiplicativeInverse
 Complex
 Real

    Head to the help> REPL then type "Real" without the double quotes. It should tell you that it is an "Abstract supertype for all real numbers". This means Real is an abstract type but also a subtype of the Number abstract type. Julia can allow this hierarchy because of how the type system was designed by its creators. You will know it's a subtype because it has the <: notation.

    Head to the julia> REPL then run the following code.

    subtypes(Real)
    Plain text

    It should give you this output. They are still all abstract types!

    4-element Vector{Any}:
 AbstractFloat
 AbstractIrrational
 Integer
 Rational

    Okay, let's skip a bit since this is going to become too nested. Let's run the following code instead.

    supertypes(Int64)
    Plain text

    It should give you this output.

    (Int64, Signed, Integer, Real, Number, Any)

    The supertypes method checks the supertype (opposite of subtype) of a concrete or an abstract type. Int64 is a concrete type. So the first supertype of Int64 is Signed which is an abstract type, then the supertype of Signed is Integer which is also an abstract type. Are you seeing a pattern?

    All types in Julia, including all abstract types and concrete types, are a subtype of the abstract type Any which signifies "any kind of type". This also means all types in Julia ultimately inherit from it. The diagram below should help you out.

    But the examples only give you a glimpse of how the type system works with abstract types and concrete types. It's still not clear what the differences between the two are yet unless you try to write some code.

    In the next step, you will learn how to write abstract types and concrete types so that it can be explained further.

  • Step 4: Part 1 | Building Your Shape Types

    Let's start building your own Shape package and module. What does your Shape module do? It will contain the following functionalities and features:

    • Contains custom abstract types for specific polygons under an AbstractPolygon type. 
    • Contains custom abstract types that inherit the AbstractPolygon type. 
    • Contains concrete types under abstract types that are polygon specific e.g. Quadrilateral, and AbstractTriangle.

    Let's write this one in code. First, head to the root directory of your project (from Quest 1 / Tutorial 1) and open a new terminal. Run the following command on your terminal using the shell.

    julia

    This should open a julia> REPL. Don't close it yet. In the julia> REPL, run the following code.

    import Pkg
Pkg.generate("Shape.jl")
    Plain text

    This should create a new directory called Shape.jl within the root directory of your project. Take a look at the screenshot below so you can see what other files it generates.

    Head into the Shape.jl directory. Head to the src directory and open the file Shape.jl. Once opened, you will see that it already contains some code.

    On the same julia> REPL, run the following code.

    Pkg.activate("Shape.jl")
    Plain text

    This should create a similar output like "Activating project at `~/unleashing-julia/Shape.jl`". To ensure that you really activated the Shape.jl environment, run the following code in the same julia> REPL.

    using Shape; Shape.greet();
    Plain text

    This should output "Hello World!". If that is the case, that means it is working. If that is not the case, then check that your julia> REPL is at the root directory of your project by closing it and opening a new one at the root directory of your project.

    Close the julia> REPL as you won't need it at this point.

    In your Shape.jl file, replace all of its contents with the following code.

    module Shape

abstract type AbstractShape end
abstract type Polygon <: AbstractShape end
abstract type Irregular <: AbstractShape end

end # module Shape
    Plain text

    You can see here that the code has a root abstract type of AbstractShape with its corresponding subtypes that are also abstract types Polygon and Irregular. The latter is named like so because you can have shapes that do not have a clearly defined shape, an irregular shape.

    Remember that the <: symbol denotes that the left-hand side is a subtype of the right-hand side.

    Your focus for this example is building the subtypes for Polygon abstract type. Let's write an abstract type for four sided polygons, a Quadrilateral.

    Now, let's write the concrete type for Square. Here is the code for constructing a concrete type Square.

    abstract type Quadrilateral <: Polygon end  # abstract type for four-sided polygons

struct Square <: Quadrilateral
	len::Float64
	Square(len::AbstractFloat) = len < 0 ? error("Length is less than 0.") : new(convert(Float64, len))
	Square(len::Integer) = len < 0 ? error("Length is less than 0.") : new(convert(Float64, len))
end
    Plain text

    With this code, you can see that unlike abstract types, concrete types have a structure such as properties and sometimes an inner constructor method. The property you see here is the len field for the length of a square's sides.

    There are also two inner constructor methods where each one is assigned to argument types under the abstract type AbstractFloat and abstract type Integer. Other types not under those two abstract types will cause a "MethodError" error.

    A constructor method constructs an object, in this case, an object of type Square. This means the Square type is only created if, and only if, the value passed to create the object is not less than 0.

    Add the following line just below the module Shape line.

    export Square
    Plain text

    This will export the name Square to the namespace whenever you are importing the module into your code or in the julia> REPL. The full code by now will look like this.

    module Shape

export Square

abstract type AbstractShape end
abstract type Polygon <: AbstractShape end
abstract type Irregular <: AbstractShape end

abstract type Quadrilateral <: Polygon end  # abstract type for four-sided polygons

struct Square <: Quadrilateral
	len::Float64
	Square(len::AbstractFloat) = len < 0 ? error("Length is less than 0.") : new(convert(Float64, len))
	Square(len::Integer) = len < 0 ? error("Length is less than 0.") : new(convert(Float64, len))
end

end # module Shape
    Plain text

    Open a new julia> REPL at the root directory of your project. Run the following code.

    import Pkg
Pkg.activate("Shape.jl")
    Plain text

    Run each line of code below and observe.

    using Shape
Square(10)
Square(1.2)
Square(-1)
    Plain text

    The last line will error with the following output.

    ERROR: Length is less than 0.
Stacktrace:
 [1] error(s::String)
   @ Base ./error.jl:35
 [2] Square(len::Int64)
   @ Shape ~/unleashing-julia/Shape.jl/src/Shape.jl:14
 [3] top-level scope
   @ REPL[9]:1

    You can also assign a variable as well. Write the code below to the julia> REPL.

    square = Square(10)
println(square.len)
    Plain text

    It should output "10.0". This means, the variable square also now has the field name .len from the Square object it was constructed from.

  • Step 5: Part 2 | Building Your Shape Types

    Let's write a concrete type for Rectangle. Write the following code to your Shape.jl file and just below the definition of the Square concrete type.

    struct Rectangle <: Quadrilateral
	len::Float64
	width::Float64
	function Rectangle(len::T, width::Z) where T <: Union{AbstractFloat, Integer} where Z <: Union{AbstractFloat, Integer}
    	len < 0 && return error("Length is less than 0.")
    	width < 0 && return error("Width is less than 0.")
    	new(convert(Float64, len), convert(Float64, width))
	end
end
    Plain text

    You have discovered the use of the where keyword. It's used to assign the types that are aliased in the arguments, in this case, the len and width have a type alias called T and Z. The syntax, T <: Union{AbstractFloat, Integer}, signifies that T represents a type which is an alias for the union of AbstractFloat and Integer types. The same could be said to the Z, it's also an alias for the union of AbstractFloat and Integer types.

    The keyword Union lets you construct union types, which are used for values that can only have a few specific types. For example, a value that could be either an AbstractFloat or an Integer would use a union type.

    Just below the export Square line, add the following code to Shape.jl.

    export Rectangle
    Plain text

    Close all your terminals including the REPLs. Open a new julia> REPL at the root directory of your project. Run the following code on the julia> REPL.

    import Pkg 
Pkg.activate("Shape.jl")
using Shape
Rectangle(1,1)
Rectangle(12.5, 10)
Rectangle(1,-1)
    Plain text

    The last should error that the width is less than 0.

    Let's create a new function named "area" to put what we've learned into practice. The area of a square is calculated by squaring its side, s, where s is the side's length. A rectangle's area is calculated by multiplying its length and width.

    On the same julia> REPL, write the following code.

    area(shape::Square) = shape.len ^ 2
area(shape::Rectangle) = shape.len * shape.width
methods(area)
    Plain text

    The last line should have a similar output below.

    # 2 methods for generic function "area" from Main:
 [1] area(shape::Rectangle)
 	@ REPL[5]:1
 [2] area(shape::Square)
 	@ REPL[4]:1

    That means, your area function has two methods, one for the Square concrete type and one for the Rectangle concrete type. Let's test by creating a square and a rectangle as variables using the concrete types on the same julia> REPL.

    square = Square(10)
rectangle = Rectangle(10, 10)  # A square is just a rectangle
area(square)
area(rectangle)
    Plain text

    Both should output a value of 100.0 since the area of a square is derived from the area of a rectangle since a square is a rectangle but with sides of the same length.

    At this point, you get the gist of how to create a hierarchy of abstract types and concrete types in Julia.

    One thing to be careful about is that you cannot create subtypes for concrete types as they are not abstract types. Only abstract types can create subtypes of either abstract types or concrete types. For example, changing

    struct Rectangle <: Quadrilateral
	len::Float64
	width::Float64
	function Rectangle(len::T, width::Z) where T <: Union{AbstractFloat, Integer} where Z <: Union{AbstractFloat, Integer}
    	len < 0 && return error("Length is less than 0.")
    	width < 0 && return error("Width is less than 0.")
    	new(convert(Float64, len), convert(Float64, width))
	end
end
    Plain text

    to the following code

    struct Rectangle <: Square
	len::Float64
	width::Float64
	function Rectangle(len::T, width::Z) where T <: Union{AbstractFloat, Integer} where Z <: Union{AbstractFloat, Integer}
    	len < 0 && return error("Length is less than 0.")
    	width < 0 && return error("Width is less than 0.")
    	new(convert(Float64, len), convert(Float64, width))
	end
end
    Plain text

    will lead to this error as shown in the following screenshot after reinitialising the Shape module in another julia> REPL.

    It means that Rectangle can only be subtyped by an abstract type since it is a concrete type.

  • Step 6: Stacking It Up!

    Congratulations! You have finished this short tutorial about multiple dispatch, and a glimpse towards how its type system works!

    Feel free to extend your Shape module and refactor it, for example, add an AbstractTriangle type and extend from there. You can start by replacing your Shape.jl code with the following code snippet.

    module Shape

export Square
export Rectangle
export area

abstract type AbstractShape end
abstract type Polygon <: AbstractShape end
abstract type Irregular <: AbstractShape end

abstract type Quadrilateral <: Polygon end

struct Square <: Quadrilateral
	len::Float64
	Square(len::AbstractFloat) = len < 0 ? error("Length is less than 0.") : new(convert(Float64, len))
	Square(len::Integer) = len < 0 ? error("Length is less than 0.") : new(convert(Float64, len))
end

struct Rectangle <: Quadrilateral
	len::Float64
	width::Float64
	function Rectangle(len::T, width::Z) where T <: Union{AbstractFloat, Integer} where Z <: Union{AbstractFloat, Integer}
    	len < 0 && return error("Length is less than 0.")
    	width < 0 && return error("Width is less than 0.")
    	new(convert(Float64, len), convert(Float64, width))
	end
end

area(shape::Square) = shape.len ^ 2
area(shape::Rectangle) = shape.len * shape.width

end # module Shape
    Plain text
