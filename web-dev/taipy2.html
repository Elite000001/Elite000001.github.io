  • Step 1: Understanding Scenarios in Taipy

    Taipy Scenarios allow you to model, execute, and compare different data workflows, making them a powerful tool for decision-making. A Scenario in Taipy is a collection of Data Nodes and Tasks arranged in a Directed Acyclic Graph (DAG). Here's how the key components work:

    Data Nodes: represent a variable in Taipy. Data Nodes do not contain the data itself but point to the data and know how to retrieve it. These Data Nodes can point to different types of data sources like CSV files, databases, etc., and they can represent various types of Python variables such as integers, strings, data frames, lists, and more. They are fully generic and can be used to represent datasets, parameters, KPIs, intermediate data, or any variable.

    Tasks: are the translation of functions in Taipy where their inputs and outputs are data nodes.

    Scenarios: Scenarios are created by combining Data Nodes and Tasks to form a graph that maps the execution flow. Each scenario can be submitted, resulting in the execution of its tasks. End-users often require the modification of various parameters to reflect different business situations. Taipy provides the framework to execute various scenarios under different conditions (i.e. various data/parameter values set by end-users). When possible, Taipy runs the tasks in parallel.

    Scenarios provide features like:

    • Managing pipeline executions with minimal effort.
    • Avoiding redundant computations by detecting unchanged data.
    • Supporting concurrent execution for improved scalability.

    Configuration: is a structure to define scenarios. It serves as the blueprint for our Directed Acyclic Graph(s) and models the data sources, parameters, and tasks. After being defined, a configuration functions like a superclass and is employed to generate various instances of scenarios.

    Time to create our first data nodes, these nodes can represent input, intermediate, or output data for tasks. To create data nodes Taipy provides a method called configure_data_node present inside the Config class. Add the following code after the filter_company_names function, around line 67: 

    # Data Nodes for country and company names

    The id parameter in the configure_data_node method represents a variable.  In this case, it's country and company_names, well see how easy it is to create data nodes 😉.

    Similar to the configure_data_node method, for tasks we use the configure_task method present in the Config class. Add the following code after the company_names_cfg variable, around line 70:

    filter_companies_task_cfg = Config.configure_task(

    output=company_names_cfg,
    function=filter_company_names
)

    The function parameter points to our function filter_companies_names , which takes country as an argument and returns the companies from that country,  and it takes a country_cfg as an input node and companies_names_cfg as the output node.

  • Step 5: Creating Scenarios

    Scenarios are the collections of Data Nodes and Tasks representing workflows that can be executed, modified, and compared. In other words, it groups related tasks together to achieve a specific goal. To create a scenario, we will use the configure_scenario method. Add the following code after filter_companies_task_cfg, around line 76:

    scenario_cfg = Config.configure_scenario(
    id="country_company_scenario",
    task_configs=[filter_companies_task_cfg]
)
    Python

    With Config.configure_scenario, we define a new scenario configuration. Once configured, the scenario can be instantiated and executed.

    Here you can see task_configs is a list, meaning we can execute multiple tasks in a single scenario. The filter_companies_task_cfg task filters company names based on the selected country and writes the result to a data node. Here's how you can visualize a scenario with a task in it.

  • Step 6: Handling Callbacks

    Callbacks in Taipy provide a dynamic mechanism to respond to user actions or data changes. By defining callback functions, we can update data nodes, trigger tasks, and refresh scenarios in real-time. This step demonstrates how to implement callbacks for an interactive and responsive application. Add the following code after the scenario_cfg, around line 80.

    def on_change(state, name, value):
    if name == "country":
        print(f"{name} was modified to {value}")
        # Update the country data node in the scenario
        state.scenario.country.write(value)
        # Submit the scenario to execute the task
        state.scenario.submit(wait=True)
        # Read the updated company names and update the state
        state.company_names = state.scenario.company_names.read()
    if name == "company":
        print(f"{name} was modified to {value}")
    Python

    In this step, the on_change callback listens for changes to the country or company selection. When a country is modified, it updates the associated data node, submits the scenario for processing, and retrieves the filtered company list for the GUI. However, if the name is "company", it simply logs the change to the console. This integration ensures that the backend logic and frontend interface remain in sync, delivering a smooth user experience.
